Sometimes there can be literally thousands of blocks in a single Scratch sprite. This causes an immense amount of lag and stuttering of performance while doing very simple activities including scrolling around the code area, dragging blocks, searching for blocks, typing, running the project, etc. Sometimes the project may outright crash and cause you to lose progress if you don’t constantly download the project every few minutes. Ideally, code should be spread out between multiple sprites if they do different tasks to minimize this. However, that is not always possible if you are working on a large, advanced remixed project and/or complex project with other people. Simply splitting up the code area into multiple separate sections/tabs that you can individually click on would be a great way to reduce this problem by a lot. Each section can roughly be a set amount of blocks that you pick based on your device’s performance like 1,000 or 2,000 blocks. The code for splitting up the code doesn’t have to be intricate by any means and could just be arbitrary as to where they get split because most of the time you will be working on one section at a time only. The amount of time it takes to occasionally switch between these sections to make changes would be much less than the wasted time of always slowly lagging and losing your progress coding.
TL;DR
- Thousands of blocks in a Scratch sprite causes an overwhelming amount of lag causing many crashes and wastes your time. This could be greatly improved by splitting up the code area arbitrary into manageable independent sections. This will improve performance and allow for more stability so you can peacefully code.
