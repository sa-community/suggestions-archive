When (for example) sending a message, scratch has to decide which codeblock to execute first.
After some experimenting, it seems like scratch always executes the codeblock first, which headblock is the oldest, tho sometimes it does something different. It is important to note, that this pattern (seems) to always be consistent, even after a reload or downloading and re-uploading the project.
It might be possible to change this execution order by modifying the internal creation order, a timestamp or however it is stored to give more control over your scratch project.

By message, do you mean broadcasts? If so, the execution order is very well defined: the code is executed in the order of the sprites and clones' layers: ie. The sprite on front executes first.
If you're referring to multiple "when I receive broadcast" at the same sprite, yes, the order is unexpected and unintuitive.
HOWEVER, This way of coding is not recommended; try to fit all of your code under the same WIRB block. If 2 codes need time to run (for example, contain loops), you should use 2 different broadcasts.
